# Zero_Nine 后续开发阶段详细计划

**作者：Manus AI**

## 文档目的

本文将基于前一版《Zero_Nine 对齐十三层 HarnessAgent 架构的现状评估与路线图》，把后续开发工作进一步展开为一套**可直接执行的阶段计划**。重点不是再讨论“该不该做”，而是回答三个更具体的问题：**每个阶段具体做什么、先后顺序如何安排、以及怎样才算做完**。整体规划仍遵循此前已经确定的原则，即 **先把第3—4层闭环做硬，再补第5—7层协作能力，再推进第8—11层的平台与治理，最后强化第12层技能进化与横切可观测能力**。[1] [2] [3]

当前 Zero_Nine 已经具备统一 Rust workspace、Claude/OpenCode 宿主接入、Spec 工件目录、执行验证主链路、宿主自动续跑、自适应测试策略，以及远程 finish 安全确认闭环等基础能力，因此已经适合从“原型骨架”进入“阶段化产品开发”模式。[1] [2]

| 规划维度 | 本文处理方式 |
| --- | --- |
| 规划范围 | 从下一步开发开始，到形成完整十三层主干平台前的全部阶段 |
| 规划粒度 | 细化到“阶段 → 里程碑 → 任务包 → 产出物 → 验收标准” |
| 规划对象 | Rust 核心 crate、宿主适配层、CLI、工件体系、治理体系、演化模块 |
| 规划目标 | 让 Zero_Nine 后续开发可以按阶段直接排期与落地 |

## 一、总体开发顺序

Zero_Nine 接下来的开发不适合按“功能点零散补丁”推进，而应按**平台成熟度**分阶段建设。因为这个项目的核心价值不在单个命令，而在于它能否逐步成长为一个**可验证、可编排、可治理、可进化**的统一 Agent Harness。

因此，后续建议按 **六个开发阶段** 推进。它们之间是严格前后依赖关系，而不是可随意互换的并行模块。只有前一阶段把契约和状态语义做稳，后一阶段才不会建立在脆弱基础之上。

| 阶段 | 阶段名称 | 核心目标 | 对应架构层 |
| --- | --- | --- | --- |
| 阶段 1 | 结构化 Spec 与执行闭环强化 | 把现有单代理闭环做成稳定主干 | 第2、3、4、9层 |
| 阶段 2 | 状态同步与漂移检测体系 | 让系统具备现实对账与重规划能力 | 第3层 |
| 阶段 3 | 多角色编排与协作协议 | 从单链路执行器升级为编排器 | 第5、6层 |
| 阶段 4 | 任务运行器与生命周期平台 | 打通接单、执行、回写与总控视图 | 第7、8层 |
| 阶段 5 | 安全治理与人类监督体系 | 从硬编码门禁升级到策略系统 | 第10、11层 |
| 阶段 6 | 技能进化与可观测闭环 | 形成经验沉淀、复用与持续优化能力 | 第12层 + 横切层 |

## 二、阶段 1：结构化 Spec 与执行闭环强化

这个阶段是整个后续开发中最重要的一段，因为它决定 Zero_Nine 后面的所有能力是不是建立在**强合同**和**可验证闭环**之上。当前系统已经具备 proposal、tasks、context protocol、自动续跑和真实验证执行等能力，但这些能力仍偏“主干已跑通，模型仍较松”。因此，阶段 1 的目标不是扩新入口，而是把已有主链路做成**强结构、强状态、强证据**。

### 1.1 阶段目标

这一阶段结束后，Zero_Nine 应当满足三个结果。第一，Spec 不再只是文件集合，而是有明确 schema 的结构化合同。第二，任务执行不再只是“跑一下然后记日志”，而是每一步都有前置条件、执行动作、验证证据和 verdict。第三，失败不再是一句文本，而是有可计算的失败分类和后续策略。

| 项目 | 阶段目标 |
| --- | --- |
| 目标 G1 | 将 `zn-spec` 升级为强结构化 Spec 层 |
| 目标 G2 | 将 `zn-loop` 升级为小步验证驱动执行循环 |
| 目标 G3 | 将 `zn-exec` 升级为统一 evidence / verdict 产出器 |

### 1.2 任务拆解

#### 任务包 A：Spec schema 强化

这里的重点是把当前 proposal / task 的文档式表达，升级为**可约束、可校验、可扩展**的数据合同。建议在 `zn-types` 中先扩展核心结构，然后由 `zn-spec` 负责序列化、持久化和 schema 校验。

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| A1 | 扩展 Proposal 字段：目标、范围、非目标、约束、依赖、风险级别、验收标准、回滚策略 | `zn-types` |
| A2 | 扩展 Task 字段：前置条件、完成条件、验证动作、失败升级策略、责任角色 | `zn-types` |
| A3 | 在 `.zero_nine/spec/` 下引入 schema version 与 schema validator | `zn-spec` |
| A4 | 为 proposal / tasks 增加 DAG 关系表达，支持 depends_on / blocks | `zn-types` + `zn-spec` |
| A5 | 提供 CLI 校验命令，例如 `zero-nine spec validate` | `zn-cli` |

#### 任务包 B：执行循环强化

此处目标是让 `zn-loop` 不再只是顺序推进任务，而是具备**显式状态机**。建议把 task lifecycle 统一成 `Pending -> Ready -> Running -> Verifying -> Passed / Failed / Blocked / Escalated`。

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| B1 | 定义 task 运行状态机与转换规则 | `zn-types` |
| B2 | 将 `run` / `resume` 统一到同一 execution state machine | `zn-loop` |
| B3 | 每个任务执行前先检查 preconditions | `zn-loop` + `zn-exec` |
| B4 | 每个任务执行后必须进入 verification step | `zn-loop` + `zn-exec` |
| B5 | 为失败任务加入 retry / rollback / halt / escalate 规则 | `zn-loop` |

#### 任务包 C：证据与 verdict 标准化

当前验证已经有真实命令执行和证据留痕，但接下来需要把这些产物收敛为统一结构，以便后续漂移检测、平台总控、策略引擎和技能蒸馏复用。

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| C1 | 定义 `ExecutionEvidence` 统一结构 | `zn-types` |
| C2 | 定义 `TaskVerdict`：Passed / Failed / Blocked / NeedsHuman / Drifted | `zn-types` |
| C3 | 统一命令计划、实际执行命令、stdout/stderr 摘要、退出码和工件路径 | `zn-exec` |
| C4 | 生成标准化 verification report 文件 | `zn-exec` + `zn-spec` |
| C5 | 将 verdict 写回任务状态与 proposal 总结 | `zn-loop` |

### 1.3 预期产出物

| 产出物 | 形式 |
| --- | --- |
| Spec v2 schema | Rust 类型 + 持久化 schema 文件 |
| Task lifecycle 状态机 | 类型定义 + 编排逻辑 |
| Evidence / Verdict 标准 | 类型定义 + JSON/Markdown 报告模板 |
| `spec validate` 命令 | CLI 子命令 |
| 阶段设计说明 | `docs/spec_v2.md` |

### 1.4 验收标准

| 编号 | 验收标准 |
| --- | --- |
| AC1 | 任意 proposal 都能通过 schema 校验或返回精确错误 |
| AC2 | task 必须显式包含 preconditions、verification 和 completion 条件 |
| AC3 | `run` / `resume` 产出的每个 task 均有标准 verdict |
| AC4 | 执行报告能清楚显示“计划动作、实际动作、证据、判定” |
| AC5 | 旧项目工件在迁移后仍可被读取或被引导升级 |

### 1.5 依赖关系与风险

阶段 1 是后续所有阶段的基础，因此不应被压缩。这里最大风险是 schema 设计过快定死，导致第3阶段多角色协作时又要返工。解决办法是：**先把状态和约束语义做稳，再把协作角色字段预留出来，但不要过早引入复杂协议。**

## 三、阶段 2：状态同步与漂移检测体系

阶段 2 的目的，是让 Zero_Nine 不再只会“按计划做”，而会在执行前、执行中和执行后检查**现实世界是否仍然允许这样做**。这正是第3层“状态同步与漂移检测”的核心价值。[1]

### 2.1 阶段目标

这一阶段的完成标志，是 Zero_Nine 拥有一套**Desired State / Actual State / Diff / Response Policy** 机制。它不仅能发现项目状态偏离，还能根据偏离类型决定继续执行、局部重规划、请求人工确认，还是直接停止。

| 项目 | 阶段目标 |
| --- | --- |
| 目标 G4 | 建立期望状态与现实状态模型 |
| 目标 G5 | 建立漂移 diff engine |
| 目标 G6 | 在关键入口统一接入漂移检测 |

### 2.2 任务拆解

#### 任务包 D：状态模型定义

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| D1 | 定义 `DesiredProjectState`：分支、依赖、关键文件、测试入口、远程能力 | `zn-types` |
| D2 | 定义 `ActualProjectState` 采集模型 | `zn-types` + `zn-exec` |
| D3 | 在执行前采集 repo、workspace、toolchain、auth 状态 | `zn-exec` |
| D4 | 在 Spec 中记录关键期望状态 | `zn-spec` |

#### 任务包 E：diff engine 与阈值策略

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| E1 | 定义 `StateDiff` 结构与 diff 分类 | `zn-types` |
| E2 | 增加差异级别：Info / Warning / Blocking / Dangerous | `zn-types` |
| E3 | 为不同 diff 级别定义默认策略 | `zn-loop` |
| E4 | 输出漂移报告并写入 runtime events | `zn-spec` + `zn-loop` |

#### 任务包 F：关键入口接线

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| F1 | `run` 前执行 drift preflight | `zn-loop` |
| F2 | `resume` 前执行 drift re-entry check | `zn-loop` |
| F3 | `finish-branch` 前执行 remote readiness check | `zn-exec` |
| F4 | 验证失败后若检测为环境漂移，自动转入 replan 路径 | `zn-loop` |

### 2.3 预期产出物

| 产出物 | 形式 |
| --- | --- |
| 状态采集器 | Rust 模块 |
| 漂移比较引擎 | Rust 模块 |
| 漂移报告 | `.zero_nine/runtime/drift_report.*` |
| 状态差异策略表 | `docs/drift_policy.md` |

### 2.4 验收标准

| 编号 | 验收标准 |
| --- | --- |
| AC6 | 在关键入口能自动检测分支、脏工作区、工具缺失、认证缺失等问题 |
| AC7 | diff engine 能输出结构化差异结果，而非单纯字符串日志 |
| AC8 | 系统能区分“可继续”“需重规划”“需人工确认”“必须阻止”四类反应 |
| AC9 | 漂移导致的失败可与普通执行失败明确区分 |

### 2.5 依赖关系与风险

阶段 2 强依赖阶段 1 的强结构化 Spec 和标准 verdict。如果前一阶段没有完成，这一阶段的 diff 结果将缺乏参照物。其主要风险是状态采集过度膨胀，因此应从最小必需状态集开始，不要一次把所有环境信息都采进来。

## 四、阶段 3：多角色编排与协作协议

阶段 3 才是 Zero_Nine 从“强执行器”迈向“真正 Agent 编排器”的起点。当前系统只有单主链路，因此很多复杂任务仍会把规划、执行、验证混在一起。这个阶段的目标，是先在本地单机范围内，把这些角色拆开。

### 3.1 阶段目标

| 项目 | 阶段目标 |
| --- | --- |
| 目标 G7 | 引入角色化任务编排 |
| 目标 G8 | 定义多角色协作消息协议 |
| 目标 G9 | 防止多角色任务扩散失控 |

### 3.2 角色模型建议

建议第一版只实现三个固定角色：**Planner、Executor、Reviewer**。Planner 负责将目标转成 Spec 和 DAG 任务。Executor 负责执行具体动作。Reviewer 负责验证结果、指出缺陷、触发返工或放行。

| 角色 | 主要职责 | 输入 | 输出 |
| --- | --- | --- | --- |
| Planner | 规划、拆解、设置验收标准 | 用户目标、上下文协议 | 结构化 Spec、Task DAG |
| Executor | 实施代码或工具动作 | Task、工作区、上下文协议 | 修改结果、执行证据 |
| Reviewer | 复核结果、给出 verdict | 执行证据、任务验收标准 | 审查 verdict、返工建议 |

### 3.3 任务拆解

#### 任务包 G：角色化编排

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| G1 | 定义 `AgentRole`、`AgentSession`、`Assignment` 模型 | `zn-types` |
| G2 | 在 `zn-loop` 中实现 role-aware scheduler | `zn-loop` |
| G3 | 每个角色使用独立上下文与工件命名空间 | `zn-spec` + `zn-loop` |
| G4 | 为 Executor / Reviewer 准备独立工作区 | `zn-exec` |

#### 任务包 H：协作协议

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| H1 | 定义消息类型：PlanReady、TaskAssigned、ExecutionDone、ReviewReturned | `zn-types` |
| H2 | 定义冲突解决策略 | `zn-loop` |
| H3 | 定义 role handoff 审计记录 | `zn-spec` |
| H4 | 增加跨角色上下文压缩协议 | `zn-loop` + `zn-host` |

#### 任务包 I：终止与补偿

| 子任务 | 说明 | 主要改动模块 |
| --- | --- | --- |
| I1 | 定义全局 budget：最大轮数、最大失败次数、最大返工次数 | `zn-types` |
| I2 | 超限后执行 halt / escalate / summarize | `zn-loop` |
| I3 | 定义补偿动作：回滚、撤销、重新规划 | `zn-loop` + `zn-exec` |

### 3.4 预期产出物

| 产出物 | 形式 |
| --- | --- |
| 角色模型 | Rust 类型 |
| 多角色调度器 | `zn-loop` 子模块 |
| 协作协议定义 | 文档 + 类型模型 |
| 多角色运行时工件 | `.zero_nine/runtime/agents/` |

### 3.5 验收标准

| 编号 | 验收标准 |
| --- | --- |
| AC10 | 一个复杂任务可被拆分为 Planner / Executor / Reviewer 三段链路 |
| AC11 | 各角色切换有清晰消息事件，不依赖隐式共享上下文 |
| AC12 | 返工次数、失败次数与终止条件可配置 |
| AC13 | 多角色链路失败时，系统能生成完整总结而不是悬空状态 |

### 3.6 依赖关系与风险

这一阶段高度依赖前两阶段的 Spec 与 diff 体系。其最大风险是过早做成“伪分布式系统”。建议第一版只做**单机多角色、异步事件风格、非分布式部署**，先把协作语义跑顺。

## 五、阶段 4：任务运行器与生命周期平台

阶段 4 的目标，是让 Zero_Nine 从“对着本地项目运行”进一步升级到“能接收工作项、完成闭环并回写交付”。这对应教材中的第7层与第8层。[1]

## 六、阶段 5：安全治理与人类监督体系

阶段 5 的意义，是把当前已经存在的远程 finish 显式确认，从一个单点开关扩展为**统一治理系统**。因为只要多角色编排、任务运行器和远程回写开始广泛使用，安全治理就会从“可选项”变成“生死线”。[4]

### 5.1 阶段目标

| 项目 | 阶段目标 |
| --- | --- |
| 目标 G13 | 定义统一风险等级与授权矩阵 |
| 目标 G14 | 建立审批票据模型 |
| 目标 G15 | 支持人工接管与驳回 |

| M2 | 定义风险等级：Low / Medium / High / Critical | `zn-types` |
| M3 | 建立默认授权矩阵 | `zn-loop` + `zn-exec` |
| M4 | 将关键动作统一走 policy check | `zn-exec` |

| --- | --- | --- |
| N1 | 设计 `ApprovalTicket` 数据结构 | `zn-types` |
| N2 | 将 `--confirm-remote-finish` 升级为通用确认机制 | `zn-cli` + `zn-loop` |
| N3 | 为被阻断动作生成审批指引与恢复命令 | `zn-host` + `zn-cli` |
| N4 | 提供人工接管、驳回、重规划命令 | `zn-cli` |

#### 任务包 O：审计与治理报告

| 子任务 | 说明 | 主要改动模块 |

| --- | --- |

### 5.4 验收标准

| 编号 | 验收标准 |
| --- | --- |
| AC18 | 所有高风险动作必须经过统一 policy check |
| AC19 | 高风险动作被阻止时，系统能生成明确恢复路径 |
| AC20 | 人工可显式批准、驳回、接管或要求重规划 |
| AC21 | 所有审批与策略判定都能追溯 |

### 5.5 依赖关系与风险

阶段 5 建立在阶段 4 的生命周期平台之上，因为没有生命周期和总控视图，治理会变得割裂。其最大风险是策略系统过于复杂，因此建议先做**本地内建 policy model**，把接口预留好，再视需要考虑兼容 OPA 等外部策略引擎。[4]

## 七、阶段 6：技能进化与可观测闭环

阶段 6 是 Zero_Nine 的长期壁垒所在。只有当系统能从自己的执行历史中提炼模式、积累技能，并通过观测体系知道这些技能是否真的提高了成功率，Zero_Nine 才真正进入“闭环进化平台”的状态。

### 6.1 阶段目标

| 项目 | 阶段目标 |

| 目标 G17 | 从运行历史中自动蒸馏技能候选 |
| 目标 G18 | 建立基本观测与评估体系 |

### 6.5 依赖关系与风险

阶段 6 必须放在最后，因为没有前面稳定的 Spec、evidence、verdict、policy 和 lifecycle，就没有足够可靠的数据去蒸馏技能。最大风险是过早自动化技能注入，导致系统“学坏”。因此第一版建议坚持**候选生成自动化、启用决策半自动化**。

## 八、跨阶段通用工程要求

虽然上面按阶段拆分，但有几类工作必须从第一阶段开始就始终贯彻，否则后面代价会很大。

### 8.1 契约优先

所有新增能力都应优先落在 `zn-types` 的明确模型上，再进入 `zn-spec`、`zn-loop`、`zn-exec` 与 `zn-cli`。不要让运行逻辑先于类型契约随意膨胀，否则第3阶段以后会非常难维护。

### 8.2 工件优先于隐藏状态

Zero_Nine 的一个核心优势是其工件目录和可追踪运行记录。因此新增能力都应优先考虑“是否能被写成工件、是否能被回放、是否能被审计”，而不是只存在于内存逻辑中。

### 8.3 保持目录克制

你之前对项目交付一直强调**结构简洁、避免膨胀**。因此每个阶段都要优先复用现有 crate 边界，除非真的出现职责无法承载的情况，否则不要轻易拆出过多新 crate。新增目录应尽量克制，文档、工件和模块都要放在准确位置，避免形成后期治理负担。

| 通用要求 | 执行原则 |
| --- | --- |
| 类型契约 | 先改 `zn-types`，再改运行逻辑 |
| 工件可追溯 | 所有关键状态迁移都应落盘 |
| 目录简洁 | 不因局部功能过度拆分 crate |
| 向后兼容 | 旧工件可读取，或可自动迁移 |
| 先 dry-run 再高风险动作 | 所有远程与破坏性动作都应有预览路径 |

## 九、建议的实际排期方式

如果你准备正式开干，我建议采用“**阶段开发 + 每阶段 2 到 3 个冲刺**”的节奏。每个阶段都先做模型设计和文档，再做核心逻辑，再做 CLI 与宿主暴露，最后做编译与冒烟验证。这样不会一开始就把所有精力消耗在外部体验层，而能优先保证内核稳定。

| 阶段 | 建议冲刺数 | 推荐节奏 |
| --- | --- | --- |
| 阶段 1 | 3 个冲刺 | 先 schema，再状态机，再证据模型 |
| 阶段 2 | 2 个冲刺 | 先状态采集，再 diff 策略接线 |
| 阶段 3 | 3 个冲刺 | 先角色模型，再调度器，再终止补偿 |
| 阶段 4 | 2 个冲刺 | 先任务导入回写，再总控视图 |
| 阶段 5 | 2 个冲刺 | 先风险矩阵，再审批与人工接管 |
| 阶段 6 | 3 个冲刺 | 先技能模型，再蒸馏，再评估体系 |

## 十、你现在最应该立即启动的开发包

如果你希望我给出一个最现实的开工顺序，那么我建议你立刻把开发聚焦在下面三个开发包上，因为它们分别是后续所有阶段的“地基、刹车和分工入口”。

| 优先级 | 建议立即启动的开发包 | 原因 |
| --- | --- | --- |
| P1 | 阶段 1 的任务包 A：Spec schema 强化 | 所有后续能力都依赖强结构化合同 |
| P2 | 阶段 2 的任务包 D + E：状态模型与 diff engine | 让系统有现实校验能力，避免盲跑 |
| P3 | 阶段 3 的任务包 G：角色化编排 | 为从单代理走向多角色系统开门 |

从工程视角看，**先把 Spec 做硬、再把漂移检测做实、再把角色编排引入**，是最稳妥也最有复利的路线。只要这三步走对，Zero_Nine 后面无论是平台化、治理化还是技能化，都会顺很多。
