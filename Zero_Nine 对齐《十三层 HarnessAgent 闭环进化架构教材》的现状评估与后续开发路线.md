## 一、这份十三层教材为什么适合作为 Zero_Nine 的主线蓝图

教材把系统分成 **12 个功能层 + 1 个横切可观测层**，并特别强调三件事：第一，系统必须从“会执行”升级到“会验证地执行”；第二，系统必须从“能派工”升级到“有协作协议”；第三，系统必须把日志和经验沉淀为可复用技能资产[1]。这三点，正好对应 Zero_Nine 目前最需要继续拉高的方向。

Zero_Nine 当前 README 已明确把项目定位为一个以 Rust 编写的统一编排内核，试图整合 OpenSpec、Superpowers、Ralph-loop 与 OpenSpace 的语义边界，并通过统一数据模型、状态机、工件目录与宿主适配层，向 Claude Code 与 OpenCode 暴露单一工作流入口[2]。这说明项目从一开始就不是“写几个脚本把模型串起来”，而是朝着**可治理的 Agent Harness** 去做的。因此，用你发来的教材去约束后续开发，不是“换方向”，而是把原本已经形成的技术取向正式升格为架构宪法。

> 简单说，**Zero_Nine 现在已经有了“发动机舱”和“传动系统”，而这份教材可以用来规定整车该怎么继续长成一台真正可控、可持续、可进化的车。**

| 教材主张 | Zero_Nine 当前对应情况 | 说明 |
| --- | --- | --- |
| 第1层是可替换的编码 Agent 商品层 | 已有 `zn-host` + 宿主适配导出 | Claude/OpenCode 已接入统一命令接口[2] |
| 第3—4层是执行闭环核心 | 已有 `zn-loop` + `zn-exec`，并补上宿主自动续跑、真实验证与硬门禁 | 这是当前最成熟的一段 |
| 第9层是规格工具层 | 已有 `zn-spec`、proposal、tasks、progress、context protocol | 但还未完全成为强结构化 Spec 平台 |
| 第10—11层是治理闭环 | 仅完成远程 PR/merge 显式确认门禁的早期形态 | 还不是完整策略引擎 |
| 第12层是技能进化层 | 已有 `zn-evolve` 骨架 | 但还没有真正的技能仓库与自动蒸馏闭环 |

## 二、Zero_Nine 现在到底进展到什么程度

从目前仓库结构看，Zero_Nine 已经形成了一个清晰的 Rust workspace，包含 `zn-types`、`zn-spec`、`zn-exec`、`zn-loop`、`zn-evolve`、`zn-host` 与 `zn-cli` 七个核心 crate，并且 README 已将这些模块分别对应到共享数据模型、工件管理、执行策略、循环调度、演化候选、宿主适配与命令入口等职责[2]。这说明它已经越过了“概念验证”阶段，至少进入了**可编译、可运行、可迭代的架构原型期**。

更关键的是，最近一轮收尾改造已经把三个原来最容易停留在“概念层”的点推进到了“有真实执行语义”的状态。第一，宿主 Brainstorm 在会话达到 `Ready` 后已经可以沿同一条 `run` 链路自动续跑到 proposal 绑定、Spec 校验与执行阶段；第二，验证阶段已经会根据仓库特征自动选择 Rust、Python、Node 对应测试命令，并把真实解析命令写入证据文件；第三，finish-branch 中对远程 PR / merge 已经加入显式安全确认门禁，未确认时只给预览与指引，确认后才允许远程动作继续执行。这些都说明 Zero_Nine 已不再只是“会记录任务状态”，而是已经开始具备**可执行动作闭环**[3]。

| 当前已落地能力 | 状态 | 所属架构含义 |
| --- | --- | --- |
| Claude / OpenCode 宿主接入 | 已有 | 第1层商品 Agent 适配层的统一入口 |
| `.zero_nine/` proposal / tasks / progress / runtime events | 已有 | 第2层运行时 + 第9层规格工件雏形 |
| 宿主 Brainstorm Ready 后自动续跑到执行 | 已有 | 第4层验证驱动执行主链路雏形[3] |

| 结构化上下文注入协议 | 已有 | 第2层上下文容器与运行时协议雏形 |
| 按项目类型自适应测试 | 已有 | 第4层真实验证器 |
| finish-branch 远程显式确认 | 已有 | 第10—11层最早期治理门禁 |
| `zn-evolve` 演化候选模块 | 已有骨架 | 第12层入口，但尚未形成强闭环 |

## 三、把当前进度映射到十三层：你现在站在哪里

如果按照教材的分层逻辑来评估，Zero_Nine 当前最强的不是“层数覆盖面”，而是**中下层闭环已经开始成形**。也就是说，项目并不是每层都做了一点，而是已经在第1层、第2层、第3—4层和第9层之间打出了一条从输入目标到可执行任务的主路径。

更具体地说，第1层已经相当清楚，因为 Claude Code 与 OpenCode 已被当作可替换宿主接入。第2层已经有基础形态，因为工具执行、工件目录、上下文协议、运行状态记录都已经出现。第3层虽然还没有完整的“期望状态 vs 实际状态”对账引擎，但已经开始通过 Spec 门禁、状态恢复与工作区检测承担现实校验的部分职责。第4层则是当前完成度最高的区域，因为验证驱动执行、自动续跑、测试证据、失败留痕和工作区回退都已经开始真实工作。第9层也已经有雏形，因为 proposal、tasks、progress 与结构化 context protocol 已经在承担“自然语言目标 → 机器可执行工件”的转译功能[2] [3]。

但与此同时，第5—8层和第10—12层还远没有完全展开。当前还没有真正意义上的多 Agent 协议、任务运行器、统一平台层、策略引擎、审批矩阵和技能包仓库。因此，如果严格按教材衡量，Zero_Nine 目前的成熟度更接近：**“完成了第1—4层与第9层之间的一条单代理闭环主干，并在第10层和第12层上打了预埋点。”**

## 四、现在最应该怎么理解“接下来一步步开发”

你现在最需要避免的，是一看到 13 层就开始同时铺 13 条线。教材自己已经给出一个非常重要的建设顺序：**先接入，后闭环；先闭环，后协作；先协作，后平台；先平台，后治理；最后再做长期进化**[1]。这句话对 Zero_Nine 非常适用。

结合当前进度，我建议你把接下来的开发拆成 **四个大阶段、九个具体里程碑**。这样既能沿着教材前进，又不会把当前已经形成的主干打散。

| 大阶段 | 目标 | 核心原因 |
| --- | --- | --- |
| 阶段A：把第3—4层做硬 | 先把单代理闭环真正做稳 | 没有稳定闭环，多 Agent 和平台层只会放大错误 |
| 阶段B：补第5—7层协作与接单能力 | 从“单链路执行器”升级到“可分工系统” | 这是从工具走向系统的分水岭 |
| 阶段C：补第8—11层平台与治理 | 从“能跑”升级到“可组织化使用” | 否则远程动作和多任务运行风险会迅速上升 |
| 阶段D：做第12层与横切观测 | 从“能完成任务”升级到“越跑越强” | 这是长期护城河所在 |

## 五、分阶段开发路线：最推荐的落地顺序

### 第一阶段：把当前闭环主干从“可用”做成“稳定”

这一阶段应只围绕第2—4层与第9层继续打磨，不要急着上多 Agent。当前最有价值的工作，是把已有的自动续跑、验证证据、上下文协议、Spec 绑定和 finish-branch 主链路全部做成**可重复、可回放、可诊断**。

第一，你要把 `zn-spec` 从“proposal / tasks 文件夹”升级为真正的**结构化 Spec 合同层**。这意味着 proposal 不应只是文档式状态，而应有明确字段，比如目标、范围、非目标、依赖、约束、验收标准、风险等级、执行策略和回滚策略。第二，你要把 `zn-loop` 中的执行循环进一步补成**小步验证驱动**，让每个 task 具备前置条件、完成条件、失败分类、重试次数和升级条件。第三，你要把 `zn-exec` 的验证结果标准化，形成统一的 `plan -> action -> evidence -> verdict` 数据结构。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M1 | 把 proposal 升级为强结构化 Spec，补 DAG、验收标准、风险级别 | 第9层 |
| M2 | 在执行循环中引入失败分类学：环境漂移、工具错误、验证失败、策略阻断、人类否决 | 第3—4层 |
| M3 | 统一验证证据模型与任务 verdict 模型，支持重试 / 回滚 / 停机条件 | 第4层 |

### 第二阶段：补上真正的状态同步与漂移检测

教材里第3层强调“现实校验”，这是 Zero_Nine 当前还没有真正做实的一块[1]。现在虽然有门禁与工作区回退，但还缺“计划世界”和“现实世界”的系统性对账。因此，第二阶段要做的不是新功能，而是为已有功能加上**不盲目前进的能力**。

这一阶段建议新增一个独立的 drift 子模块，用来收集项目实际状态，例如当前分支、工作区脏状态、关键文件是否存在、依赖是否可用、测试入口是否存在、远程认证是否完整、上一步工件是否齐备，然后与 Spec 和任务计划做 diff。只要 diff 超出阈值，就不继续执行，而是进入 replan 或人工确认。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M4 | 增加 Desired State / Actual State 模型 | 第3层 |
| M5 | 增加 diff engine 与阈值策略 | 第3层 |
| M6 | 在 run / resume / finish-branch 前统一接入漂移检测 | 第3—4层 |

### 第三阶段：从单代理闭环走向多角色编排

当前 Zero_Nine 最容易继续扩展的地方，是在已有 `zn-loop` 基础上把“单主链路”拆成“多角色子任务”。这一步不要一开始就做复杂分布式系统，而应先在本地做三个固定角色：**Planner、Executor、Reviewer**。其中 Planner 负责规格拆解，Executor 负责代码与工具动作，Reviewer 负责验证与审查意见。每个角色先在独立上下文和独立工作区运行，然后通过事件协议交换结果。

这一步做成之后，Zero_Nine 才会真正从“带宿主的执行器”升级为“Agent 编排器”。教材特别强调第5层和第6层的区别：第5层解决调度，第6层解决协作协议[1]。所以这里应先做编排，再做协议，不要一步跳太大。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M7 | 在 `zn-loop` 内引入角色化任务编排与独立工作区 | 第5层 |
| M8 | 定义子 Agent 之间的消息契约、状态同步与冲突合并规则 | 第6层 |
| M9 | 引入全局终止条件与补偿动作，避免多任务无限扩散 | 第5—6层 |

### 第四阶段：把任务接单、交付回写和平台视角补齐

当第5—6层站稳之后，才值得做第7层和第8层。第7层不是简单接 Jira，而是做“任务编译器”；第8层不是简单做界面，而是把需求、规格、执行、验证、审批与交付装入统一生命周期[1]。对 Zero_Nine 来说，这一步可以先从 GitHub Issue / PR 生态开始，而不是一上来就接很多外部系统。

也就是说，你可以先做一个最小可用任务运行器：读取 GitHub Issue 模板或本地 issue 文件，映射成 Zero_Nine 的 proposal/spec，执行后回写 PR、comment、status summary。之后再把这些状态做成一个轻量 Web 面板或 TUI，总览需求、执行、验证、审批和演化候选。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M10 | 做最小 GitHub Issue → Proposal → PR 回写闭环 | 第7层 |
| M11 | 补生命周期状态机：Draft / Ready / Running / Review / Approved / Completed | 第8层 |
| M12 | 做统一总控视图，至少先提供 TUI 或静态 dashboard | 第8层 + 横切层 |

### 第五阶段：把治理层从“单个确认开关”升级为策略系统

现在的 `--confirm-remote-finish` 很重要，但它只是一个早期治理门禁，不足以代表第10—11层已经完成。下一步应该从“硬编码确认参数”升级到**风险分级 + 审批矩阵 + 策略规则**。例如，读取、写文件、运行测试、创建分支、推送远程、开 PR、合并、删除资源，应拥有不同风险级别与不同默认策略。

这一阶段最合适的技术路线，是先在 Rust 内做本地策略引擎的简单版本，再考虑是否接入 OPA 之类外部策略系统。因为教材明确把 OPA 视为第10层典型参考[1]，但你没必要一开始就把完整外部策略引擎引入。先把 Zero_Nine 自己的 policy model 设计清楚，更重要。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M13 | 定义动作风险等级与默认授权矩阵 | 第10层 |
| M14 | 把 `--confirm-remote-finish` 扩展为通用审批票据模型 | 第10—11层 |
| M15 | 提供人工接管、人工驳回、人工重规划接口 | 第11层 |

### 第六阶段：最后再做技能蒸馏与经验回流闭环

教材反复强调，第12层是长期护城河[1]。对 Zero_Nine 而言，这一层不该只是 `zn-evolve` 产出几个候选文件，而应该真正形成“日志 → 模式提炼 → 技能包 → 版本管理 → 可控复用”的完整闭环。换句话说，你最终想要的不是“系统这次做完了”，而是“系统下一次对类似任务默认更聪明”。

这一阶段建议先从最朴素的方式做起：把成功任务和失败任务各自归档，抽取适用场景、前置条件、关键命令、失败征兆、禁止条件，生成版本化 skill bundle。随后，再允许用户在 run 时显式启用某个 skill bundle，最后再考虑自动推荐。

| 里程碑 | 要做什么 | 对应层 |
| --- | --- | --- |
| M16 | 定义 skill bundle 结构：适用范围、前置条件、禁用条件、风险等级、版本号 | 第12层 |
| M17 | 从 runtime events / verification evidence / reports 中自动蒸馏候选技能 | 第12层 + 横切层 |
| M18 | 支持 skill bundle 注入执行链路，并跟踪复用效果 | 第12层 + 第2层 |

## 六、如果你现在就要开工，最值得优先做的前三步

如果你不想看完整路线，而是想知道“明天开始先做什么”，那我会给你一个更聚焦的答案。

第一步，不要再扩新命令，先把 **Spec 数据结构**做强。因为当前所有后续能力——漂移检测、验证驱动、多 Agent 协作、审批矩阵、技能复用——都依赖一个足够强的结构化合同层。如果 Spec 还是半文档式的，后面的系统都会不稳。

第二步，做一个真正的 **失败分类与状态对账引擎**。因为现在 Zero_Nine 最需要补的是“为什么停、为什么重试、为什么要人工介入”的清晰语义。没有这层，后面平台化时你会看不懂系统为什么这样决策。

第三步，才是把当前单链路执行拆成 **Planner / Executor / Reviewer** 三角色。因为这是第5层编排器的最小实现，也是从“单 Agent 工具”跨到“多 Agent 系统”的第一道坎。

| 优先级 | 下一步动作 | 预期结果 |
| --- | --- | --- |
| P1 | 强化 `zn-spec` 的结构化合同模型 | 为所有上层能力打底 |
| P2 | 做漂移检测 + 失败分类学 | 让系统知道为什么停、何时重规划 |
| P3 | 做三角色本地编排 | 让 Zero_Nine 从单链路走向协作式架构 |

## 七、我对你现在阶段的总体判断

如果用一句话总结，我会这样说：**Zero_Nine 现在已经走过了“只是想法”的阶段，也已经走过了“只有一层宿主适配”的阶段，当前正处在“第1—4层主干开始成形，但距离完整十三层平台还有明显工程鸿沟”的关键窗口期。**

这其实是一个很好的阶段。因为它说明你不是从零开始，也说明你还没有被错误的平台复杂度拖住。现在最重要的，不是追求看起来完整，而是沿着你这份教材规定的顺序，把每一层之间的契约、状态、验证和治理语义一点点做硬。只要顺序对，Zero_Nine 是完全可以沿着这份 Markdown 继续发展成你要的那种 **Harness Agent 闭环进化平台** 的。
