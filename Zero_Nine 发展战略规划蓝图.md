## 零、架构核心思想

> **模型提供能力，流程提供秩序，策略提供边界，进化提供未来。**

> **模型决定能力上限，架构决定系统下限；工具决定起步速度，分层决定长期成败。**

十三层 Harness Agent 闭环进化架构的核心不是层数本身，而是补上了可靠系统最关键的三个缺口：
1. **执行闭环缺口** → 第 4 层 任务规划与验证驱动执行层
2. **协作协议缺口** → 第 6 层 多 Agent 协作层
3. **长期进化缺口** → 第 12 层 技能进化与经验复用层

## 第一部分：战略定位与愿景

### 1.1 核心愿景

> **把 Zero_Nine 打造成 AI 原生开发的"操作系统级"Harness 平台——让 Agent 从"一次性对话工具"进化为"可持续、可治理、可进化的工程伙伴"。**

### 1.2 价值主张

| 维度 | 行业现状痛点 | Zero_Nine 解决方案 |
|-----|------------|-----------------|
| **需求管理** | 对话即失，无法追溯 | OpenSpec 工件持久化 + 版本化 Spec 合同 |
| **执行约束** | 无质量门禁，代码质量不稳定 | Superpowers 技能链 + TDD 验证闭环 |
| **任务恢复** | 中断后从头开始 | Ralph-loop 状态机 + 断点恢复 |
| **经验积累** | 每次任务从零开始 | OpenSpace 技能演化 + 经验资产库 |
| **协作治理** | 单 Agent 黑盒执行 | 多角色编排 + 策略引擎 + 人类监督 |

### 1.3 架构宪法

Zero_Nine 遵循十三层 Harness Agent 架构的分层原则：

```
模型提供能力 → 流程提供秩序 → 策略提供边界 → 进化提供未来
```

**四层核心职责**：
- **需求层 (zn-spec)**: 工件与知识管理
- **执行层 (zn-exec)**: 实现与验证
- **调度层 (zn-loop)**: 循环与恢复
- **进化层 (zn-evolve)**: 观测与演化

**横切关注点**：
- **可观测性**: Trace / Metrics / Logs / Evals 全局贯穿

**五大教学板块**（理解架构的正确方式）:

| 板块 | 对应层级 | 核心问题 | 升级意义 |
|-----|---------|---------|---------|
| 基础执行板块 | 第 1—2 层 | 系统有没有可靠行动能力 | 解决底层能力接入与底座统一 |
| 闭环控制板块 | 第 3—4 层 | 系统能不能持续做对 | 解决漂移检测与验证驱动执行 |
| 协作交付板块 | 第 5—8 层 | 系统能不能规模化交付 | 解决多 Agent 协作与工程流转 |
| 治理决策板块 | 第 9—11 层 | 系统能不能受控可信 | 解决规格、策略与人类拍板 |
| 进化智能板块 | 第 12 层 + 横切层 | 系统能不能越用越强 | 解决经验沉淀与群体复用 |

### 2.2 Zero_Nine 当前成熟度评估

| 层级 | 教材目标 | Zero_Nine 现状 | 成熟度 | 优先级 |
|-----|---------|--------------|-------|-------|
| **横切层** | Trace/Metrics/Logs/Evals | 有 events.ndjson、ExecutionReport，但无统一观测面板 | 🟡 早期 | P2 |
| **第 1 层** | 可替换 Agent 资源池 | Claude/OpenCode 已接入统一入口 | 🟢 较强 | - |
| **第 2 层** | 工具执行与记忆 | 工件与上下文已具备，长期记忆需增强 | 🟡 中等偏早 | P1 |
| **第 3 层** | 对账、漂移、重规划 | 有门禁与部分环境校验，缺 diff engine | 🟡 早中期 | P0 |
| **第 4 层** | 分步、验证、重试、回滚 | 最成熟，真实验证 + 自动续跑+worktree | 🟢 当前最强 | - |
| **第 5 层** | 调度、派工、隔离、终止 | 已有 loop，但单主链路为主 | 🟡 早期 | P1 |
| **第 6 层** | 协议、一致性、事务 | 缺口较大 | 🔴 缺失 | P2 |
| **第 7 层** | Issue→Agent→PR 回写 | 缺口较大 | 🔴 缺失 | P2 |
| **第 8 层** | 生命周期、质量门、审批流 | 早期设想，未产品化 | 🔴 缺失 | P2 |
| **第 9 层** | 结构化 Spec、DAG、验收 | 已有 proposal/task，DAG 与强约束需加强 | 🟡 中等 | P0 |
| **第 10 层** | Policy as Code、最小权限 | 只有远程 finish 显式确认 | 🟡 早期 | P1 |
| **第 11 层** | 裁决、接管、最终审批 | 已有"确认后执行"雏形 | 🟡 早期 | P1 |
| **第 12 层** | 技能蒸馏、版本化、复用 | zn-evolve 骨架，缺资产闭环 | 🟡 骨架期 | P2 |

### 2.3 核心结论

> **Zero_Nine 当前成熟度定位**：
> - ✅ **第 1—4 层与第 9 层**之间已打通一条**单代理闭环主干**
> - ⚠️ **第 5—8 层**（协作与平台）和**第 10—12 层**（治理与进化）仍偏早期
> - 🎯 **下一步策略**：先把第 3—4 层做硬，再补第 5—7 层，再上第 8—11 层，最后做第 12 层强化

## 第三部分：分阶段发展路线图

### 3.1 总体阶段划分

```
阶段 A (2026 Q2)    阶段 B (2026 Q3)    阶段 C (2026 Q4)    阶段 D (2027 Q1)
把第 3—4 层做硬   →   补第 5—7 层    →   补第 8—11 层   →   做第 12 层与横切
单代理闭环稳定       从单链路到协作       从能跑到可组织化      从完成任务到越跑越强
```

### 3.2 阶段 A：把第 3—4 层做硬（2026 年 Q2）

**目标**：先把单代理闭环真正做稳，没有稳定闭环，多 Agent 和平台层只会放大错误。

#### 里程碑 M1：结构化 Spec 合同层强化

| 任务 | 描述 | 验收标准 | 负责模块 |
|-----|------|---------|---------|
| M1-1 | Proposal 数据结构增强 | 增加 `scope_in`、`scope_out`、`constraints`、`risks` 字段 | `zn-types` |
| M1-2 | DAG 依赖验证增强 | 实现循环依赖检测、关键路径分析 | `zn-spec` |
| M1-3 | 验收标准结构化 | `acceptance_criteria` 从文本改为可验证条目 | `zn-types` |
| M1-4 | Spec 验证报告 | 生成 `spec-validation.json`，包含错误/警告分级 | `zn-spec` |

#### 里程碑 M2：失败分类学与状态对账

| 任务 | 描述 | 验收标准 | 负责模块 |
|-----|------|---------|---------|
| M2-1 | Desired/Actual State 模型 | 定义计划世界与现实世界的对账结构 | `zn-types` |
| M2-2 | Drift Diff Engine | 实现文件差异、分支状态、依赖可用性检测 | `zn-exec` |
| M2-3 | 失败分类学 | 环境漂移/工具错误/验证失败/策略阻断/人类否决 | `zn-types` |
| M2-4 | 漂移检测门禁 | 在 run/resume/finish-branch 前统一接入检测 | `zn-loop` |

#### 里程碑 M3：验证证据模型标准化

| 任务 | 描述 | 验收标准 | 负责模块 |
|-----|------|---------|---------|
| M3-1 | EvidenceRecord 结构 | `plan → action → evidence → verdict` 数据链 | `zn-types` |
| M3-2 | Verdict 模型 | Passed/Failed/Blocked/Escalated 状态机 | `zn-types` |
| M3-3 | 重试/回滚策略 | 基于失败分类的自动重试与回滚条件 | `zn-loop` |
| M3-4 | 停机条件 | 超过阈值自动 Escalated 并通知人工 | `zn-loop` |

### 3.3 阶段 B：补第 5—7 层协作与接单能力（2026 年 Q3）

**目标**：从"单链路执行器"升级到"可分工系统"，这是从工具走向系统的分水岭。

| M4-4 | 独立工作区隔离 | 每个角色在独立 worktree 上下文运行 | `zn-exec` |

#### 里程碑 M5：多 Agent 协作协议（第 6 层）

| 任务 | 描述 | 验收标准 | 负责模块 |

| 债务项 | 影响 | 偿还优先级 | 预计工作量 |

| `zn-exec` 空心化 | 执行层多是骨架，真实调用宿主代理逻辑待完善 | P0 | 3 人周 |
| 技能演化落地难 | evaluations.jsonl 和 candidates/格式有了，但真正的技能注入和回滚机制未实现 | P2 | 4 人周 |
| 宿主依赖耦合 | 高度依赖 Claude Code/OpenCode 的插件生态，平台政策变化有风险 | P1 | 持续监控 |
| 复杂度门槛 | 1290+ 行类型定义 + 四层抽象，新贡献者学习曲线陡峭 | P1 | 文档改进 |
| 无云端同步 | 本地工件模式适合个人，团队协作需要云端状态共享 | P3 | 暂不处理 |

### 5.2 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|-----|------|------|---------|

| 执行层持续空心化 | 高 | 高 | 优先完成真实执行闭环，发布 MVP |
| 市场竞争加剧 | 高 | 中 | 聚焦差异化：工件驱动 + 可恢复 + 可审计 |
| 用户学习曲线陡峭 | 中 | 中 | 改进文档，提供 demo_project 体验路径 |
| 技能演化闭环难产 | 中 | 高 | 先从朴素方式做起，再考虑自动推荐 |
